## R3 で観察された「scope creep の予兆 / 微増」 ⚠

ただし、以下のファイルは件数が単調上昇 / 微増しており、scope creep の予兆として **次ラウンドで動向を観察** すべきです:

### docs/07-runbook-templates/manual-audit-runbook.md (5 → 7 → 9 単調上昇)

- R3 で 7 件の P2 が出た
- 主因: ea0ee20 で paste 5 軸 / Viewport 3 軸を Appendix A に隔離したことで、§2 簡略化と §5 必須ゲート (6→5 項目に統合) と Appendix A の整合確認が一気に発生
- これは **構造再編に伴う一時的な指摘増**。次 R4 で減衰すれば正常、減衰しなければ scope cut 検討

### examples/workshop-course1-snippet.md (3 → 3 → 4 微増)

- R3 で 3 件 P2 が出た
- 主因: docs/02/04 との数値整合 / scope cut 4 条件との対応 / R1→R2 の v2 部分適用説明など、**説明追加要求系**
- 「説明を増やす」要求は scope creep のサイン (`docs/04` の修正の負債化判定項目に該当)

## 各ラウンドで適用していた監査軸

R3 で件数増加した原因の分析として、各ラウンドの軸が変わっていることも記録:

| ラウンド | 監査軸の重点 |
|---|---|
| R1 | 公式仕様 / 内部整合 / リンク / PASS-FAIL 分離 / 文書品質 (基本軸) |
| R2 | R1 軸 + 「R1 反映で新たな矛盾が入っていないか」 |
| R3 | R2 軸 + **構造再編後の整合 + コア/adapter 階層 + N=1 仮説の誠実性 + 内部参照の明記** |

R3 で「構造再編整合」を最重視したため、ea0ee20 の構造再編で漏れた整合不足が大量検出された (P1×6 = ほぼ全部この系統)。これも件数増加の一因。

## 教訓

### 1. 件数推移は **同一ファイルでの同一監査軸** で比較する

ファイル数 / 監査軸が変わると件数は当然変わる。「総件数下降」は範囲を固定して測定する必要がある。本ツールキットの収束判定 (`docs/04`) も「直近 2-3 ラウンドの総件数が下降傾向」と書いているが、**範囲が固定されている前提**を明示するべき (`docs/04` 改善余地)。

### 2. 大きな構造変更 (構造再編 / 新規 docs 追加) の直後は件数が一時的に増える

ea0ee20 のような大規模構造再編の直後の R3 は、整合性チェックが大量発生するため、件数増加が **scope creep ではなく構造再編の必然的後始末** であることが多い。R4 で減衰するかが正念場。

### 3. ファイル別単調上昇は scope creep 予兆として個別に追跡

総件数だけでなく **各ファイルの推移** を見て、単調上昇しているファイルは scope creep 予兆として個別判定する。本リポでは docs/07/manual (5→7→9) が最も注意を要する。

### 4. 範囲拡大時は新規ファイル件数を別計測

R3 のように範囲拡大した場合、「新規ファイルの初回件数」と「旧ファイルの推移」を分けて計測する。両者を混ぜて総件数で見ると判断を誤る。
