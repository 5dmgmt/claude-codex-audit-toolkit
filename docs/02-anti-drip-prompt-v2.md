# 02. 五月雨防止プロンプト v2 (実走証拠ベース)

## 位置づけ

このドキュメントは **SIFT Phase C ランブックを 13 ラウンドかけて R13 で ALL PASS に到達させた本物の codex プロンプト構造** をテンプレ化したものです。

> ⚠ **v1 の重圧版は廃止しました**。「全行 enumerate 強制 / mandatory セクション / 3 段階構造 / 全カテゴリ均等深さ」の 4 要素は、Codex CLI 0.125.0 を tool 呼び出しループに陥れて finding 出力前に session exit する原因と判明しました。詳細は [`08-known-pitfalls.md`](08-known-pitfalls.md) 参照。

## 実走で効いた 4 つの構造要素

### 要素 1. 冒頭 1 行で監査対象 + 軸数を明示

```
{ランブック名} ({ランブック相対パス}) を以下 N 軸で網羅的に監査してください。
```

- 1 ラウンドで監査対象を **1 ファイル** に絞る (これが Codex tool ループ防止の最重要)
- 関連参照ファイルが必要なら「必要なら Read」と書いて強制しない
- 軸の数 N は 5-7 個が実走の目安 (SIFT は 7 軸 / Workshop は 6 軸)

### 要素 2. 五月雨防止 4 行ブロック + pre-condition 蓄積リスト

```
【五月雨防止プロンプト】 — 過去ラウンドとの矛盾 / 同一違和感の深堀り cascade を禁止。
- 過去ラウンドで反映済の修正を pre-condition として受け入れる (例: {蓄積リスト})。これらを「再考すべき」として再指摘しない
- 同じ違和感の角度を変えた言い換え / 表現の揺れを別件として再指摘しない (1 違和感 = 1 件で集約)
- 1 ラウンドで全 P1/P2/P3 を重複なく列挙し、後続ラウンドでの新規発見を禁止する前提で網羅
- 矛盾する指摘は出す前に「前ラウンド方針との整合」を自己検証してから出す。それでも出すなら明示的に「前ラウンド方針 X との矛盾承知の上で」と注記
```

**最重要ポイント**: 1 行目の「(例: {蓄積リスト})」に **過去ラウンドで反映済の具体的方針を 30+ 項目蓄積して書く**。これが矛盾指摘 cascade を実質的に止めていた本物の仕掛けです。

SIFT R13 で実走した蓄積リストの一部 (抜粋):

```
過去ラウンドで反映済の修正を pre-condition として受け入れる (例:
§10 新設、scope cut ルール訂正、3 階層判定、P1×0+未承認 P2×0 統一、
Phase 本体実スキーマ整合 (id/title 不可侵)、
deliverableTemplate と prompt 判定軸分離、母数 255/206、
例外承認 5 項目、heredoc 化、4 領域表記、五月雨防止プロンプト、
sift-last-sync 追加、Step 8a (起動前 .env.local 検査)、
横スクロール tab、3 Course × 5 動作明示、next-env.d.ts 厳密判定、
commit pin + __SHA__ 置換、§10.7.1 進捗マスター §10.3/§10.6 行追加、
Setup タブ実体整合チェック、§5.7 全 worktree clean 検査 + untracked、
... [30+ 項目続く]
)。これらを「再考すべき」として再指摘しない
```

R1 では「初回」と書き、R2 以降は前ラウンドで反映した修正を箇条書きで蓄積していきます。**蓄積を端折ると矛盾指摘 cascade が再発** します。

### 要素 3. 監査軸を 5-7 個、各軸 1-2 行で簡潔記述

```
【監査軸 N 軸】
1. 再現性: {1-2 行の検査内容}
2. PASS/FAIL の分離: {1-2 行の検査内容}
3. スコープ整合: {1-2 行の検査内容}
4. {本ランブック固有の整合軸}: {1-2 行}
5. 環境系: memory `feedback_audit_runbook_lint_checklist.md` の 14 項目を必ず読んで全項目評価
6. 設計判断: {1-2 行}
7. {追加軸 (任意)}: {1-2 行}
```

OK 例 / NG 例 / よくある誤検出 の三段例は **不要**。各軸 1-2 行のシンプル説明で十分。Codex は文脈から判定基準を読み取れます。

### 要素 4. 出力形式を 5 行構造に固定

```
【出力形式】
[P1/P2/P3] [§番号 / 行番号] タイトル
詳細: ...
修正案: ...

最後に [総合判定: ALL PASS or P1×N + P2×N + P3×N] を 1 行で。
```

ID / 検査軸名 / 矛盾チェック / 集計サマリ / 検査済セクション一覧 などの追加フィールドは **不要**。実走では 5 行構造で完走できていました。

## v1 (重圧プロンプト) 廃止理由

v1 で過剰に追加していた **4 要素** は Codex CLI 0.125.0 で実用上動かないことが分かりました。これら 4 要素を廃止し、加えて「監査対象は 1 ファイル」という派生運用原則を v2 で確定しました。

**廃止 4 要素**:

- ❌ **「全行 enumerate 強制」** — Codex が真面目に全 N 行を tool 呼び出しで読みに行く
- ❌ **「mandatory セクション (空セクション禁止)」** — Codex が形式を満たすために tool 呼び出しを延々続ける
- ❌ **「3 段階構造 (検査計画 → 本検査 → 集計)」** — 構造維持で出力負荷が膨大化
- ❌ **「全カテゴリ均等深さ」** — 各軸ごとに探索を強制 = tool 呼び出しが累積し session 枯渇

**派生運用原則** (4 要素廃止と一緒に確認された別の禁止事項):

- ⚠ **「全 N ファイル走査」** — 監査対象を多ファイルに広げると Codex が tool ループから抜け出せない。1 ラウンド = 1 ファイルに絞る

実証データは [`08-known-pitfalls.md`](08-known-pitfalls.md) 参照。
