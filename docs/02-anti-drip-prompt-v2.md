# 02. 五月雨防止プロンプト v2

## 問題の本質

Claude Code でランブックを書いて Codex で監査すると、ラウンドごとに finding が小出しに出続ける現象が起きます。1 ラウンドで全件出すべきものが、3-13 ラウンドに分散して出てくる。

**これは Codex の問題ではなく、Claude Code 側のプロンプト設計の問題です**。

検査軸が抽象的、重複排除を許容、全行 enumerate 強制が弱い、過去ラウンド反映が pre-condition として明示されていない、など複数の原因が重なります。

## 解決策: R2 以降のプロンプト冒頭に下記ブロックを必ず配置

```
【五月雨防止プロンプト】 — 過去ラウンドとの矛盾 / 同一違和感の深堀り cascade を禁止。
- 過去ラウンドで反映済の修正を pre-condition として受け入れる。これらを「再考すべき」として再指摘しない
- 同じ違和感の角度を変えた言い換え / 表現の揺れを別件として再指摘しない (1 違和感 = 1 件で集約)
- 1 ラウンドで全 P1/P2/P3 を重複なく列挙し、後続ラウンドでの新規発見を禁止する前提で網羅
- 矛盾する指摘は出す前に「前ラウンド方針との整合」を自己検証してから出す。それでも出すなら明示的に「前ラウンド方針 X との矛盾承知の上で」と注記
```

## 7 要素の詳細

### A. 全行 enumerate 強制

「網羅検証」と書くだけでは Codex が無意識に重複排除します。**全行を § 番号順に 1 行ずつ走査** を明示。

```
ランブック全 N 行を § 番号順に 1 行ずつ走査し、各行の番号 / 行番号 / 違和感 を表形式で全件出してください。「省略」「代表例のみ」「同型のため 1 件にまとめる」は禁止です。
```

### B. 重複排除 OFF 明示

「省略禁止」と書いても、Codex は同型 finding を 1 件にまとめがち。**全箇所列挙** を強制。

```
同型の問題が複数箇所にあっても全箇所列挙してください。1 件にまとめずに別 finding として出します (同型 finding にも個別の修正対象行が必要なため)。
```

### C. 検査軸を具体例付きで

各検査軸に **OK 例 / NG 例 / よくある誤検出** を 1-2 行で明示すると、Codex の判定深さが揺れません。

```
A1. 公式仕様整合 (Opus / Claude モデル名)
  OK 例: "Opus 4.7 (Claude 4.X 系)" / "claude-opus-4-7"
  NG 例: "Opus 4.6" / "claude-opus-4-6" / "最新の Claude モデル" (バージョン未特定)
  よくある誤検出: "Anthropic 公式" 表記そのものは NG ではない
```

### D. 出力件数下限を設定

「上限なし」と書くだけだと、5 件で「PASS」と出してくるケースがあります。**下限**を明示:

```
予想件数: 前回 R1 で X 件発見、本 R2 でも同等 (5-15 件) の検出能力を期待します。検出件数が極端に少ない (3 件以下) 場合は再検査してください。
```

### E. セクション別 mandatory

「全項目」と書くだけだと一部セクションを省略する。**該当なしの場合も「該当なし」と書く** を強制:

```
§0-§N + 横断 + 網羅検証完了 で計 (N+2) セクション。各セクションで最低 1 行は明示記載してください (該当なしの場合も「該当なし」と書く / 空セクション禁止)。
```

### F. 検査計画 → 本検査 → 集計の 3 段階構造

進捗報告を強制すると、Codex が途中で「もう網羅した」と判断して打ち切る現象を防げます:

```
1. 最初に各セクションの検査計画 (1 行 / 何をチェックするか) を出してから本検査開始
2. 検査中も「§X 完了」を逐次報告
3. 最後に集計サマリ (P1/P2/P3 件数 + 検査済セクション一覧)
```

### G. 1 ラウンド = 全カテゴリ均等深さ

ラウンドごとに「公式仕様 / フィールド整合 / 安全性」のどれかが軽くなる現象を防ぐ:

```
本 Round で「公式仕様整合」「フィールド間整合」「安全性」のいずれかを軽く済ませて他に集中することは禁止です。全カテゴリを R1 と同じ深度で検査してください。前回検出済の defect 種別も継続してチェックしてください。
```

## 効果実証

### SIFT Phase C ランブック (13 ラウンドで ALL PASS)

| Round | 結果 | 五月雨防止プロンプト |
|---|---|---|
| R1 | P2×多数 | 未適用 |
| R2 | P2×多数 | 未適用 |
| R3 | P2×13 / P3×5 | 未適用 (混乱期 / 矛盾指摘 cascade) |
| R4 | P2×5 / P3×1 | **v2 導入** で大幅減 (P2: 13→5) |
| R5-R12 | 着実減少 | v2 継続適用 |
| R13 | **ALL PASS** | 13 ラウンドで安定収束 |

### Workshop Course 1 ランブック (4 ラウンドで scope cut)

| Round | 結果 | 備考 |
|---|---|---|
| R1 | P1×1 / P2×35 / P3×4 = 40 件 | 五月雨防止 v2 未導入 |
| R2 | P2×10 / P3×3 = 13 件 (-67%) | v2 部分適用 |
| R3 | P2×5 / P3×1 = 6 件 (-54%) | v2 完全適用で減衰加速 |
| R4 | P2×4 / P3×2 = 6 件 (停滞) | scope cut 判断 |

R4 で件数停滞 (R3 6 → R4 6) = scope creep の徴候と判断、v3.4 で確定終了。

## 失敗パターン (やってはいけないプロンプト)

```
[NG-1] 「網羅検証してください」+ 「省略禁止」だけ
  → Codex は「網羅した」と思い込んで重複排除する

[NG-2] 検査軸を高次概念で書く
  → 「再現性」「PASS-FAIL 分離」「scope 整合」だけだと深さが揺れる
  → 具体例 (OK / NG / 誤検出) を必ず付ける

[NG-3] 出力件数の下限を書かない
  → 5 件で「PASS」と出してくるが本当は 30 件あるケース

[NG-4] 検査計画を最初に出させない
  → Codex が途中で「もう網羅した」と判断して打ち切る

[NG-5] 過去ラウンド反映を pre-condition として明示しない
  → 過去ラウンドで反映済の修正を再指摘してくる (五月雨の主因)
```
