# 08. 既知の落とし穴 — 重圧プロンプトの罠

## 用語

このドキュメントでは混乱を避けるため以下の呼称で統一します:

- **旧 v2 (deprecated)**: 初期ツールキットで「五月雨防止プロンプト v2 = 7 要素テンプレ」として推奨していた重圧版。本ドキュメントで廃止
- **現 v2 (実走証拠ベース)**: 廃止 4 要素を取り除き、SIFT R13 ALL PASS で実走した本物の構造に書き直したもの。`docs/02-anti-drip-prompt-v2.md` の内容

## 位置づけ

旧 v2 (重圧 7 要素テンプレ) のうち、実用上動作しない要素が判明したため、その実証データを記録するドキュメント。

この記録は **「ツールキット利用者が同じ罠を踏まないため」** + 「ツールキット自身の透明性のため」の二重目的で残します。

## 観察事実と推定原因

### 観察された事実 (試行 1-5 共通)

- exit code は常に 0 (Codex 自身はエラー扱いしていない)
- 最終 assistant message が emit されない (`--output-last-message` ファイル未作成)
- ログ末尾は file-read 系 tool 呼び出しの途中で打ち切られる
- 試行 5 は enumerate 強制を削除しても発生 → 単一要素の問題ではなく複合要因

### 推定原因 (この環境・このプロンプトに限定)

旧 v2 の重圧構造を Codex 0.125.0 + xhigh に投入すると、各要素の指示が以下のように tool 呼び出しを連鎖させると推定されます:

1. 「全行 enumerate 強制」→ 対象ファイル全行を tool 呼び出しで読み込む傾向
2. 「mandatory セクション (空セクション禁止)」→ 形式維持の確認に追加 tool 呼び出し
3. 「3 段階構造 (検査計画 → 本検査 → 集計)」→ 構造を満たすために重複 tool 呼び出し
4. 「全カテゴリ均等深さ」→ 各軸ごとに探索を強制される
5. 「全 N ファイル走査」(派生) → 関連ファイルを次々に読みに行く

これらが累積して session の budget / context を消費し、最終 synthesis 前に session が exit すると推測されます。**ただし試行 6 は reasoning effort と prompt の両方を変えて成功しているため、この 4 要素 + 派生 1 件のうちどれが必須要因かは因果分離できていません**。確実なのは「現 v2 (4 要素削除 + 1 ファイル原則) で同じ Codex 環境が動作する」という観察結果のみです。

## 廃止した旧 v2 (deprecated) 要素

以下の 4 要素が複合すると Codex が tool ループに陥ることが分かりました。**v2 では全面削除**:

### ❌ 全行 enumerate 強制

```
ランブック全 N 行を § 番号順に 1 行ずつ走査し、各行の番号 / 行番号 / 違和感 を
表形式で全件出してください。「省略」「代表例のみ」「同型のため 1 件にまとめる」は禁止です。
```

これを書くと Codex は対象ファイルだけでなく関連ファイル全部を再帰的に読みに行きます。

### ❌ mandatory セクション (空セクション禁止)

```
§0-§N + 横断 + 網羅検証完了 で計 (N+2) セクション。各セクションで最低 1 行は
明示記載してください (該当なしの場合も「該当なし」と書く / 空セクション禁止)。
```

形式維持のための tool 呼び出しが大量発生します。

### ❌ 3 段階構造 (検査計画 → 本検査 → 集計)

```
1. 最初に各セクションの検査計画 (1 行 / 何をチェックするか) を出してから本検査開始
2. 検査中も「§X 完了」を逐次報告
3. 最後に集計サマリ (P1/P2/P3 件数 + 検査済セクション一覧)
```

進捗報告の構造維持で出力負荷が膨大化します。

### ❌ 全カテゴリ均等深さ

```
本 Round で「公式仕様整合」「フィールド間整合」「安全性」のいずれかを軽く済ませて
他に集中することは禁止です。全カテゴリを R1 と同じ深度で検査してください。
```

各軸ごとに探索を強制 → tool 呼び出し 50+ 回 → session 枯渇。

## 現 v2 で残した本物の要素

以下は実走 (SIFT R13 ALL PASS) で動いていた本物の構造で、現 v2 でもそのまま残します:

- ✅ **五月雨防止 4 行ブロック** (1 違和感 = 1 件 / 矛盾指摘 cascade 防止)
- ✅ **過去ラウンドの反映済リストを pre-condition として 30+ 項目蓄積** (これが本物の cascade 防止仕掛け)
- ✅ **監査対象の単位は系統で異なる** (Codex tool ループ防止の最重要):
  - **ランブック監査**: 1 ラウンド = 1 ファイル / primary target は 1 ファイル、関連参照は任意・有界・明示された場合のみ Read
  - **コードベース監査**: 1 ラウンド = 1 commit (リポ全体 / 複数ファイル) + AUDIT_RUNBOOK.md を毎周 Codex に渡す運用 (詳細: [`docs/07-runbook-templates/code-audit-runbook.md`](07-runbook-templates/code-audit-runbook.md))
- ✅ **監査軸 5-7 個、各軸 1-2 行のシンプル説明**
- ✅ **出力形式 5 行構造** (`[Px][§/行]タイトル/詳細/修正案 + 総合判定 1 行`)

詳細は [`02-anti-drip-prompt-v2.md`](02-anti-drip-prompt-v2.md) 参照。

## 教訓

理論化は実走を超えない。SIFT R13 / Workshop R4 で動いていた本物のプロンプト構造を、私 (ツールキット作者) が「再現性のある手順」として後付けで一般化したとき、**人間レビュアー向けの制約 (網羅性 / 形式維持 / 均等深さ)** と **auto-tool-using agent 向けの制約** を混同しました。

人間には「全行走査せよ」「形式を満たせ」が機能する一方、Codex のような自走 agent には **tool 呼び出しの引き金** になるだけで、出力負荷だけ増えて synthesis に到達しません。

ツールキットを作るときは **実走で効いた最小要素にとどめる** が鉄則です。
