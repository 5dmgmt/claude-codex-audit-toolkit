# 08. 既知の落とし穴 — 重圧プロンプトの罠

## 位置づけ

ツールキット v1 で「五月雨防止プロンプト v2 = 7 要素テンプレ」として推奨していた構造のうち、実用上動作しない要素が判明したため、その実証データを記録するドキュメント。

この記録は **「ツールキット利用者が同じ罠を踏まないため」** + 「ツールキット自身の透明性のため」の二重目的で残します。

## 検証環境

- Codex CLI: 0.125.0 (research preview)
- model: gpt-5.5
- model_reasoning_effort: xhigh / high / medium
- 検証日: 2026-04-27 JST
- 検証対象: `claude-codex-audit-toolkit` リポ自身 (commit `cbd18c6`)

## 5 回の試行と観察

| 試行 | 設定 | プロンプト構造 | 結果 |
|---|---|---|---|
| 1 | xhigh + arg passing | 重圧プロンプト v1 (17 セクション網羅 + 10 軸均等深さ) | 30 秒で session exit、prompt echo のみ、finding ゼロ |
| 2 | xhigh + stdin pipe | v1 (同上) | 数百行のファイル読込ループに入って finding 出力前に exit |
| 3 | xhigh + arg + `--output-last-message` | v1 (同上) | output-last-message ファイル未作成 = 最終 assistant message が emit されないまま exit |
| 4 | high (xhigh から下げた) + arg | v1 (同上) | 同様、ファイル読込ループで session exit |
| 5 | high + dash stdin | v1 簡略版 (10 軸列挙のみ、enumerate 強制削除) | リンク整合チェックで深掘り、finding 出力前に exit |
| 6 | medium + シンプル指示 | 「README で 3 件出して」のみ | **3 件の有効 finding 出力に成功** |

## 観察された原因 (因果分析)

v1 の重圧プロンプトを Codex 0.125.0 に投入すると:

1. Codex は「全行 enumerate 強制」を満たすため、対象ファイル全行を tool 呼び出しで読みに行く
2. 「mandatory セクション (空セクション禁止)」を満たすため、形式維持に tool 呼び出しを延々続ける
3. 「全 N ファイル走査」を満たすため、関連ファイルを次々に読みに行く
4. 「全カテゴリ均等深さ」を満たすため、各軸ごとに探索を強制される
5. 結果として **tool 呼び出しが 50+ 回** 発生、session の budget / context が枯れて最終 synthesis 前に exit

## 廃止した v1 要素

以下の 4 要素が複合すると Codex が tool ループに陥ることが分かりました。**v2 では全面削除**:

### ❌ 全行 enumerate 強制

```
ランブック全 N 行を § 番号順に 1 行ずつ走査し、各行の番号 / 行番号 / 違和感 を
表形式で全件出してください。「省略」「代表例のみ」「同型のため 1 件にまとめる」は禁止です。
```

これを書くと Codex は対象ファイルだけでなく関連ファイル全部を再帰的に読みに行きます。

### ❌ mandatory セクション (空セクション禁止)

```
§0-§N + 横断 + 網羅検証完了 で計 (N+2) セクション。各セクションで最低 1 行は
明示記載してください (該当なしの場合も「該当なし」と書く / 空セクション禁止)。
```

形式維持のための tool 呼び出しが大量発生します。

### ❌ 3 段階構造 (検査計画 → 本検査 → 集計)

```
1. 最初に各セクションの検査計画 (1 行 / 何をチェックするか) を出してから本検査開始
2. 検査中も「§X 完了」を逐次報告
3. 最後に集計サマリ (P1/P2/P3 件数 + 検査済セクション一覧)
```

進捗報告の構造維持で出力負荷が膨大化します。

### ❌ 全カテゴリ均等深さ

```
本 Round で「公式仕様整合」「フィールド間整合」「安全性」のいずれかを軽く済ませて
他に集中することは禁止です。全カテゴリを R1 と同じ深度で検査してください。
```

各軸ごとに探索を強制 → tool 呼び出し 50+ 回 → session 枯渇。

## v2 で残した本物の要素

以下は実走 (SIFT R13 ALL PASS) で動いていた本物の構造で、v2 でもそのまま残します:

- ✅ **五月雨防止 4 行ブロック** (1 違和感 = 1 件 / 矛盾指摘 cascade 防止)
- ✅ **過去ラウンドの反映済リストを pre-condition として 30+ 項目蓄積** (これが本物の cascade 防止仕掛け)
- ✅ **監査対象を 1 ファイルに絞る** (Codex tool ループ防止の最重要)
- ✅ **監査軸 5-7 個、各軸 1-2 行のシンプル説明**
- ✅ **出力形式 5 行構造** (`[Px][§/行]タイトル/詳細/修正案 + 総合判定 1 行`)

詳細は [`02-anti-drip-prompt-v2.md`](02-anti-drip-prompt-v2.md) 参照。

## 教訓

理論化は実走を超えない。SIFT R13 / Workshop R4 で動いていた本物のプロンプト構造を、私 (ツールキット作者) が「再現性のある手順」として後付けで一般化したとき、**人間レビュアー向けの制約 (網羅性 / 形式維持 / 均等深さ)** と **auto-tool-using agent 向けの制約** を混同しました。

人間には「全行走査せよ」「形式を満たせ」が機能する一方、Codex のような自走 agent には **tool 呼び出しの引き金** になるだけで、出力負荷だけ増えて synthesis に到達しません。

ツールキットを作るときは **実走で効いた最小要素にとどめる** が鉄則です。
