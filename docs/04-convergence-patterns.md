# 04. 収束判定基準 — 正常収束 vs scope creep

## 何を判定したいか

Codex 監査ループを回していて、ラウンドごとの finding 件数を見たとき:

- **このまま継続して ALL PASS まで行くべきか** (正常収束)
- **件数が減らない / 矛盾指摘が出始めた / scope cut すべきか** (scope creep)

を識別する基準です。SIFT Phase C (13R で ALL PASS) と Workshop Course 1 (4R で scope cut) の対照ケースから抽出しました。

## 重要な前提

**「3 ラウンド超えたら scope cut」はルールではありません**。経験則 / 警戒目安に過ぎません。SIFT Phase C は 13 ラウンドかけて ALL PASS に到達しました。

判断基準は件数の単調性 / 指摘の質 / 矛盾の有無で、**ラウンド数そのものではない**。

## 正常収束 (継続すべき) パターン

以下のパターンが揃っていれば、上限なしで継続できます:

### パターン 1. 件数が単調減少

```
R1: P2×13 / P3×5
R2: P2×8 / P3×3
R3: P2×5 / P3×1
R4: P2×2 / P3×0
R5: P2×1
R6: ALL PASS
```

増加 → 減少のジグザグでも OK。**全体トレンドが下降** していれば正常。

### パターン 2. 指摘の粒度が細かくなる

```
R1: 「Phase 5 のテストが構造的に不十分」(高次)
R3: 「Phase 5.2 の paste 検証で innerText 比較が抜けている」(中次)
R6: 「Phase 5.2.3 の expected_value 末尾に \n が混入」(低次)
```

抽象 → 具象に進むのは網羅検証が深まっている証拠。

### パターン 3. 矛盾指摘が出ない

R3 で「A にすべき」と指摘 → R5 で「B にすべき」(A と矛盾) が出ない。出ても **「前ラウンド方針 X との矛盾承知の上で」と注記** されている (五月雨防止プロンプト v2 が機能している証拠)。

### パターン 4. 過去ラウンドで反映済の修正を再指摘しない

R3 で反映した修正を R5 でまた指摘してきたら五月雨。逆に **完全にスキップ** しているなら正常。

### SIFT Phase C 実例

| Round | P1 | P2 | P3 | 備考 |
|---|---|---|---|---|
| R1 | 0 | 多数 | - | baseline |
| R2 | 0 | 多数 | - | 構造調整 |
| R3 | 0 | 13 | 5 | 混乱期 (五月雨防止 v2 未導入) |
| R4 | 0 | 5 | 1 | **v2 導入で大幅減** |
| R5-R12 | 0 | 着実減少 | - | v2 継続適用 |
| R13 | 0 | 0 | 0 | **ALL PASS** |

13 ラウンドかけても、件数が単調減少 / 矛盾なし / 粒度が細かくなる、を満たしていたので継続が正解でした。

## scope creep (停止すべき) パターン

以下のパターンが 1 つでも出たら scope cut を検討:

### パターン 1. 件数が停滞 / 微増

```
R3: P2×6
R4: P2×6 (停滞)
R5: P2×7 (微増)
```

これは「網羅した残り」ではなく「目線の違いで新規発見しているだけ」の徴候。**Codex の指摘軸が無限に広げられる構造的欠陥** がある可能性が高い。

### パターン 2. 矛盾指摘が出始める

R3 で「A にすべき」→ R5 で「B にすべき」→ R7 で「やはり A にすべき」のように、**ラウンドを跨いで判断が揺れる**。これはランブック側に「どちらが正解か決定不能な曖昧さ」がある証拠。

### パターン 3. 同じ違和感の言い換えが繰り返される

```
R3: 「Phase 5.2 の paste 検証が不十分」
R5: 「Phase 5.2 の貼り付け確認方法が曖昧」
R7: 「Phase 5.2 の paste 内容比較ロジックが不明確」
```

3 つとも本質は同じ。Codex が違う角度から切り込んでいるだけで、修正してもまた別角度で出てくる。**ランブック側で「Phase 5.2 paste 検証は X / Y / Z で完了とする」と判定基準を明示** しないと永久に終わらない。

### パターン 4. 修正の負債化

ラウンドを進めるほどランブックが膨張。「これ以上書き足すと受講者が読めない」状態になっているのに Codex は更に detail を要求してくる。

### Workshop Course 1 実例

| Round | P1 | P2 | P3 | 件数 | 備考 |
|---|---|---|---|---|---|
| R1 | 1 | 35 | 4 | 40 件 | baseline (五月雨防止 v2 未導入) |
| R2 | 0 | 10 | 3 | 13 件 (-67%) | v2 部分適用 |
| R3 | 0 | 5 | 1 | 6 件 (-54%) | v2 完全適用で減衰加速 |
| R4 | 0 | 4 | 2 | 6 件 (停滞) | **scope cut 判断** |

R4 で **件数停滞 (6 → 6) + 過去ラウンドで反映済の方針への指摘も混在** = scope creep の徴候。v3.4 で確定終了が正解でした。

## 判定フロー

```
ラウンド R(N) 完了
    ↓
件数を R(N-1) と比較
    ↓
   ┌─────┴─────┐
   減少           停滞 / 微増
   ↓                ↓
 矛盾指摘なし?     R(N-2) と比較
   ↓                ↓
  Yes  No        2 ラウンド連続停滞?
   ↓   ↓            ↓
 継続 注記要求      Yes  No
                    ↓    ↓
                scope cut 1 R 様子見
```

## scope cut の実行手順

stop すると決めたら:

1. **現状 commit に「v3.4 確定」のような明示的な version tag を打つ**
2. **未対応 finding を「将来検討事項」として `_review-notes.md` 等に転記**
3. **ランブック冒頭に「v3.4 確定済 / これ以降の Codex 監査は scope creep と判断して停止」と明記**

逆にこれを書かずにただ止めると、後日「あれって PASS したんだっけ？」が起きます。

## チェックリスト

監査ループ中に毎ラウンド確認:

- [ ] 件数が R(N-1) より減少している
- [ ] 矛盾指摘が出ていない (出ていても注記がある)
- [ ] 過去ラウンドで反映済の修正を再指摘していない
- [ ] 指摘の粒度が前ラウンドより細かい (or 同等)
- [ ] ランブックが膨張しすぎていない (受講者が読める範囲)

5 つ全部 ✅ → 継続。1 つでも ❌ → scope cut 検討。
