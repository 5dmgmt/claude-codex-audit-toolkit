# 04. 収束判定基準 — 正常収束 vs scope creep

## 何を判定したいか

Codex 監査ループを回していて、ラウンドごとの finding 件数を見たとき:

- **このまま継続して ALL PASS まで行くべきか** (正常収束)
- **件数が減らない / 矛盾指摘が出始めた / scope cut すべきか** (scope creep)

を識別する基準です。SIFT Phase C (13R で ALL PASS) と Workshop Course 1 (4R で scope cut) の対照ケースから抽出しました。

## 重要な前提

**「3 ラウンド超えたら scope cut」はルールではありません**。経験則 / 警戒目安に過ぎません。SIFT Phase C は 13 ラウンドかけて ALL PASS に到達しました。

判断基準は件数の **全体下降傾向 / 指摘の質 / 矛盾の有無** で、**ラウンド数そのものではない**。なお件数推移はジグザグでも全体トレンドが下降していれば正常 (厳密な単調減少は要求しない)。

## 正常収束 (継続すべき) パターン

以下のパターンが揃っていれば、上限なしで継続できます:

> **前提**: 「件数下降傾向」は **同一ファイル / 同一監査軸** で比較する前提です。範囲拡大 (ファイル数増加 / 新規ファイル追加 / 監査軸の高度化) があると件数は当然変わるので、総件数だけ見ても判断できません。範囲拡大時は (a) 旧ファイルの推移と (b) 新規ファイルの初回件数を分けて計測します。本リポ自身の self-audit 例: [`examples/self-audit-history.md`](../examples/self-audit-history.md)。

### パターン 1. 件数が全体下降傾向

```
R1: P2×13 / P3×5
R2: P2×8 / P3×3
R3: P2×5 / P3×1
R4: P2×2 / P3×0
R5: P2×1
R6: ALL PASS
```

増加 → 減少のジグザグでも OK。**全体トレンドが下降** していれば正常。

### パターン 2. 指摘の粒度が「具体行に紐づく」方向に細粒化する

```
R1: 「Phase 5 のテストが構造的に不十分」(抽象)
R3: 「Phase 5.2 の paste 検証で innerText 比較が抜けている」(中次 / 具体行に紐づく)
R6: 「Phase 5.2.3 の expected_value 末尾に \n が混入」(低次 / 具体行に紐づく)
```

抽象 → 具象 (具体行 / API / assertion に紐づく方向) に進むのは網羅検証が深まっている証拠。

> **要注意の境界**: 細粒化と「説明追加要求」は別物。具体行/API/assertion に紐づき、反映すれば defect が消える細粒化は **正常収束**。一方、仕様や挙動を変えず説明だけ増やす要求 (例: 「ここに注意書きを追加してほしい」「説明文を膨らませてほしい」) は **scope creep / 修正の負債化サイン** ([パターン 4 参照](#パターン-4-修正の負債化))。

### パターン 3. 矛盾指摘が出ない

R3 で「A にすべき」と指摘 → R5 で「B にすべき」(A と矛盾) が出ない。出ても **「前ラウンド方針 X との矛盾承知の上で」と注記** されている (五月雨防止プロンプト v2 が機能している証拠)。

### パターン 4. 過去ラウンドで反映済の修正を再指摘しない

R3 で反映した修正を R5 でまた指摘してきたら五月雨。逆に **完全にスキップ** しているなら正常。

### SIFT Phase C 実例

> 本表は件数概要のみ。R3=18 / R4=6 / R5=5 / R13=ALL PASS は記録に明示があるが、R1/R2 の総数 と R6-R12 の詳細件数および「矛盾なし / 同一 finding 再発なし / 具体行に紐づく」の質的条件は Phase C runbook / log に基づく。

| Round | 件数 | 備考 |
|---|---|---|
| R1 | 多数 (詳細件数なし) | baseline |
| R2 | 多数 (詳細件数なし) | 構造調整 |
| R3 | 18 件 | 混乱期 (五月雨防止 v2 未導入) |
| R4 | 6 件 | **v2 導入で大幅減** |
| R5 | 5 件 | v2 + 5 fixes 適用、微減 |
| R6-R12 | 増減を挟みつつ低位推移 (詳細は Phase C runbook 参照) | v2 + 5 fixes + bypass 反映期 |
| R13 | 0 件 (ALL PASS) | 安定収束 |

13 ラウンドかけても、記録上確認できる範囲で **下降傾向継続** + 矛盾なし + 同一 finding 再発なし + 未解決 finding が具体行に紐づく、を満たしていたので継続が正解でした。

## scope creep (停止すべき) パターン

以下のパターンが 1 つでも出たら scope cut を検討:

### パターン 1. 件数が停滞 / 微増

```
R3: P2×6
R4: P2×6 (停滞)
R5: P2×7 (微増)
```

これは「網羅した残り」ではなく「目線の違いで新規発見しているだけ」の徴候。**Codex の指摘軸が無限に広げられる構造的欠陥** がある可能性が高い。

### パターン 2. 矛盾指摘が出始める

R3 で「A にすべき」→ R5 で「B にすべき」→ R7 で「やはり A にすべき」のように、**ラウンドを跨いで判断が揺れる**。これはランブック側に「どちらが正解か決定不能な曖昧さ」がある証拠。

### パターン 3. 同じ違和感の言い換えが繰り返される

```
R3: 「Phase 5.2 の paste 検証が不十分」
R5: 「Phase 5.2 の貼り付け確認方法が曖昧」
R7: 「Phase 5.2 の paste 内容比較ロジックが不明確」
```

3 つとも本質は同じ。Codex が違う角度から切り込んでいるだけで、修正してもまた別角度で出てくる。**ランブック側で「Phase 5.2 paste 検証は X / Y / Z で完了とする」と判定基準を明示** しないと永久に終わらない。

### パターン 4. 修正の負債化

ラウンドを進めるほどランブックが膨張。「これ以上書き足すと判断者が読めない」状態になっているのに Codex は更に detail を要求してくる状態。

判定項目 (1 つでも該当すれば「修正の負債化」サイン):
- ランブック行数が想定読者の読了時間を超える (例: 教材なら 1 セッション分、技術 runbook なら 30 分)
- 章数 (`##` 見出し) が 15+ で章間遷移が追えない
- 追加 finding が「仕様欠陥」ではなく「説明追加要求」中心 (= ランブックが本来カバーすべきでない説明を求めている)
- 想定読者が変わってしまった (初心者向けが上級者向けレベルになった等)

該当する場合、未対応 finding を `_review-notes.md` の「将来検討事項」へ移し、本ランブックは scope cut を検討する。

### Workshop Course 1 実例

> R1 内訳 (P1×1 / P2×35 / P3×4) は当時のラウンドログから抽出した推定値。Workshop runbook 本体には総数 40 件のみ記録されている。R2-R4 の内訳は runbook で裏取り可能。

| Round | P1 | P2 | P3 | 件数 | 備考 |
|---|---|---|---|---|---|
| R1 | 1 | 35 | 4 | 40 件 | baseline (五月雨防止 v2 未導入) |
| R2 | 0 | 10 | 3 | 13 件 (-67%) | v2 部分適用 |
| R3 | 0 | 5 | 1 | 6 件 (-54%) | v2 完全適用で減衰加速 |
| R4 | 0 | 4 | 2 | 6 件 (停滞) | **scope cut 判断** |

R4 で **件数停滞 (6 → 6) + 過去ラウンドで反映済の方針への指摘も混在** = scope creep の徴候。v3.4 で確定終了が正解でした。

## 判定フロー

```
ラウンド R(N) 完了
    ↓
件数を R(N-1) / R(N-2) と比較 (3 点で評価)
    ↓
[条件 1] 直近 2-3 ラウンドで **同一ファイル / 同一監査軸換算** の件数が下降傾向? (範囲拡大時は旧範囲の推移と新規範囲の初回件数を分けて評価 / [`examples/self-audit-history.md`](../examples/self-audit-history.md) 参照)
[条件 2] 矛盾指摘なし? (注記なき矛盾は rerun 要求 / 注記あり矛盾は人間が前方針を覆すか判断)
[条件 3] 同一 finding の再発なし?
[条件 4] 未解決 finding が具体行 / 節に紐づく?
    ↓
全条件 Yes  →  継続
1 つでも No
    ↓
1 ラウンド停滞 (同数 1 回) → 様子見 (次ラウンドで決める)
2 ラウンド連続停滞 / 同一 finding 再発 / 注記なき矛盾 / 抽象的 finding 増加 → scope cut 検討
```

判定条件の重み付け:
- **「件数推移」だけで判断しない** — 上記 4 条件すべてを見る
- **矛盾指摘の扱い**: 未注記の矛盾 = rerun または無効 / 注記あり矛盾 = 人間が前方針を覆すか判断 / 覆さないなら scope creep として却下

## scope cut の実行手順 (closure 成果物を取り逃さない順序)

stop すると決めたら、**tag を最後に打つ** 順序で実行:

1. 未対応 finding を「将来検討事項」として `_review-notes.md` 等に転記 (作業ツリーで編集)
2. ランブック冒頭に「v3.4 確定済 / これ以降の Codex 監査は scope creep と判断して停止」と明記 (作業ツリーで編集)
3. ステップ 1-2 をまとめて 1 つの commit にして確定
4. **その確定 commit に対して `git tag v3.4` を打つ** (tag が closure 成果物を含む状態を指す)

順序を逆にすると tag が「scope cut 確定処理を含まない commit」を指してしまい、後日「v3.4 tag が指す commit はまだ反映途中だった」が起きます。

## チェックリスト

監査ループ中に毎ラウンド確認:

- [ ] 直近 2-3 ラウンドで **同一ファイル / 同一監査軸換算** の件数が下降傾向 (1 ラウンド停滞は様子見、2 ラウンド連続停滞で要警戒 / 範囲拡大時は分離計測)
- [ ] 矛盾指摘なし、または注記ありで人間が前方針を覆す判断ができる
- [ ] 過去ラウンドで反映済の修正を再指摘していない (同一 finding 再発なし)
- [ ] 未解決 finding が具体行 / 節に紐づく (抽象的 finding が増えていない)
- [ ] ランブックが想定読者 / 判断者の読める範囲を超えていない (対象システムの大きさそのものは判定対象外)

5 つ全部 ✅ → 継続。1 つでも ❌ → 重大度に応じて: 1 ラウンド停滞のみなら様子見、複数条件 ❌ なら scope cut 検討。
