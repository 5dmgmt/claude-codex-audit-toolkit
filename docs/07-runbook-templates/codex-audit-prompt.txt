## Codex 監査プロンプト テンプレート (実走証拠ベース v2)
##
## SIFT Phase C R13 ALL PASS で実走した構造を抽象化したもの。
##
## 使い方:
##   TARGET_SHA=$(git rev-parse --short HEAD)
##   PROMPT=$(cat docs/07-runbook-templates/codex-audit-prompt.txt)
##   PROMPT="${PROMPT//__SHA__/$TARGET_SHA}"
##   PROMPT="${PROMPT//__RUNBOOK_NAME__/SIFT 受講者目線監査 Phase C ランブック}"
##   PROMPT="${PROMPT//__RUNBOOK_PATH__/docs/SIFT-AUDIT-PHASE-C-MANUAL-RUNBOOK.md}"
##   PROMPT="${PROMPT//__PREV_FINDINGS__/初回}"  # R2 以降は反映済リストに置換
##   codex exec -s read-only -m gpt-5.5 -c model_reasoning_effort=xhigh "$PROMPT"
##
## 重要:
## - 1 ラウンドで監査対象は 1 ファイルに絞る (多ファイル走査は Codex tool ループの原因)
## - R2 以降は __PREV_FINDINGS__ に「過去ラウンドで反映済の方針」を 30+ 項目蓄積する
## - 蓄積を端折ると矛盾指摘 cascade が再発する

__RUNBOOK_NAME__ (__RUNBOOK_PATH__) を以下 N 軸で網羅的に監査してください。

【五月雨防止プロンプト】 — 過去ラウンドとの矛盾 / 同一違和感の深堀り cascade を禁止。
- 過去ラウンドで反映済の修正を pre-condition として受け入れる (例: __PREV_FINDINGS__)。これらを「再考すべき」として再指摘しない
- 同じ違和感の角度を変えた言い換え / 表現の揺れを別件として再指摘しない (1 違和感 = 1 件で集約)
- 1 ラウンドで全 P1/P2/P3 を重複なく列挙し、後続ラウンドでの新規発見を禁止する前提で網羅
- 矛盾する指摘 (前ラウンドで採用された方針を否定する形の指摘) は出す前に「前ラウンド方針との整合」を自己検証してから出す。それでも出すなら明示的に「前ラウンド方針 X との矛盾承知の上で」と注記

【監査軸】 (各軸を 1-2 行で簡潔に。OK/NG 例の三段は不要。本ランブックの性格に合わせて 5-7 軸選ぶ)

1. 再現性: 手順を上から実行して同じ結果になるか / 環境差で false FAIL しないか
2. PASS/FAIL の分離: 判定基準が決定的か / 主観表現が混在していないか
3. スコープ整合: タスク表 / 検証手順 / 完了条件 / 進捗トラッカー の対象が完全一致しているか
4. (ランブック固有) 各セクション間の整合: {本ランブック固有の整合軸を 1-2 行で}
5. 環境系: 環境系 lint 14 項目チェックリスト (docs/05-env-lint-checklist.md) を全項目評価
6. 設計判断: dev bypass / paste 動線 / 認証フロー など、構造的決定の妥当性
7. (任意) {追加軸}: {1-2 行}

【監査対象】
- __RUNBOOK_PATH__ (commit __SHA__)

【関連参照ファイル】 (必要なら Read、強制ではない)
- (関連するファイルがあれば 5-10 件まで列挙)

【出力形式】
[P1/P2/P3] [§番号 / 行番号] タイトル
詳細: ...
修正案: ...

最後に [総合判定: ALL PASS or P1×N + P2×N + P3×N] を 1 行で。
