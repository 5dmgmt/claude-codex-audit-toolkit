## Codex 監査プロンプト テンプレート (R2 以降)
##
## 使い方:
##   TARGET_SHA=$(git rev-parse --short HEAD)
##   PROMPT=$(cat docs/07-runbook-templates/codex-audit-prompt.txt)
##   PROMPT="${PROMPT//__SHA__/$TARGET_SHA}"
##   PROMPT="${PROMPT//__RUNBOOK_PATH__/docs/runbooks/foo.md}"
##   codex exec -s read-only -m gpt-5.5 -c model_reasoning_effort="xhigh" "$PROMPT" < /dev/null
##
## 置換が必要な placeholder:
##   __SHA__              — 検査対象 commit SHA (短縮形 OK)
##   __RUNBOOK_PATH__     — ランブック相対パス (例: docs/runbooks/foo.md)
##   __ROUND_NUM__        — 現在のラウンド番号 (R1 / R2 / ...)
##   __PREV_FINDINGS__    — 前ラウンドで反映済の修正サマリ (R1 では「初回」と書く)
##   __EXPECTED_COUNT__   — 予想件数 (前ラウンド件数を参考に / R1 では「5-30 件」)

【五月雨防止プロンプト】 — 過去ラウンドとの矛盾 / 同一違和感の深堀り cascade を禁止。
- 過去ラウンドで反映済の修正を pre-condition として受け入れる。これらを「再考すべき」として再指摘しない
- 同じ違和感の角度を変えた言い換え / 表現の揺れを別件として再指摘しない (1 違和感 = 1 件で集約)
- 1 ラウンドで全 P1/P2/P3 を重複なく列挙し、後続ラウンドでの新規発見を禁止する前提で網羅
- 矛盾する指摘は出す前に「前ラウンド方針との整合」を自己検証してから出す。それでも出すなら明示的に「前ラウンド方針 X との矛盾承知の上で」と注記

【検査対象】
- ランブック: __RUNBOOK_PATH__
- commit: __SHA__ (固定)
- ラウンド: __ROUND_NUM__
- 前ラウンド反映: __PREV_FINDINGS__

【検査軸】 (各軸に OK 例 / NG 例 / よくある誤検出を内蔵)

A1. 公式仕様整合 (Anthropic / Claude モデル名 / API 仕様)
  OK 例: "Opus 4.7 (Claude 4.X 系)" / "claude-opus-4-7"
  NG 例: "Opus 4.6" / "claude-opus-4-6" / "最新の Claude モデル" (バージョン未特定)
  よくある誤検出: "Anthropic 公式" 表記そのものは NG ではない

A2. フィールド間整合 (前後の文 / 前後セクションの数値・命名・前提)
  OK 例: §3 で「3 phase 構成」→ §5 で 3 phase 全部記載
  NG 例: §3 で「3 phase 構成」→ §5 で 2 phase しか記載がない
  よくある誤検出: 「3 phase」と「3 段階」の言葉揺れは A2 ではなく A4 (用語整合)

A3. 安全性 (秘密漏洩 / 認証バイパス / RLS / CSP / 権限昇格)
  OK 例: dev bypass は二重ガード + read-only 限定 + systemRole=USER 固定
  NG 例: NODE_ENV だけで bypass / write も通る / admin UI が bypass で開く
  よくある誤検出: 「.env.tpl は commit 可」は OK (テンプレなので秘密含まない)

A4. 用語整合 (用語揺れ / 表記揺れ / 単位揺れ)
  OK 例: 全文で "phase" 統一
  NG 例: §3 で "phase"、§5 で "段階"、§7 で "step" が混在
  よくある誤検出: 引用元の英語表記を残しているケースは OK (出典明記なら)

A5. 再現性 (commit SHA 固定 / 環境依存 / BSD vs GNU / port pin)
  OK 例: TARGET_SHA=__SHA__ / [[:space:]] 採用 / port 明示
  NG 例: HEAD で diff / sed -i 単独 / port 期待値書かない
  よくある誤検出: README 内の bash example は再現性軸ではなく文書軸

A6. PASS-FAIL 分離 (判定基準が決定的か / 曖昧さがないか)
  OK 例: 「innerText 完全一致」「正規表現 ^[A-Z]{3}$ にマッチ」
  NG 例: 「適切に動作」「期待通り」「自然な表記」(主観判定)
  よくある誤検出: 受講者向けの「読みやすい」記述は OK (PASS-FAIL ではない)

A7. scope 整合 (ランブックの scope と実装範囲の一致)
  OK 例: ランブック §1 で「Course 1 のみ対象」→ Course 2 への言及なし
  NG 例: ランブック §1 で「Course 1 のみ対象」→ §5 で Course 2 の手順が混入
  よくある誤検出: 「将来的に Course 2 でも適用」のような future note は OK

【検査手順 — 3 段階構造】

1. **検査計画** (最初に必ず出す)
   - 各セクションで何を検査するか 1 行ずつ書き出す
   - 「§0 〜 §N まで全て走査します」と明示

2. **本検査** (走査開始)
   - ランブック全 N 行を § 番号順に 1 行ずつ走査し、各行の番号 / 行番号 / 違和感 を表形式で全件出してください
   - 「省略」「代表例のみ」「同型のため 1 件にまとめる」は禁止です
   - 同型の問題が複数箇所にあっても全箇所列挙してください (1 件にまとめずに別 finding として出します)
   - 検査中も「§X 完了」を逐次報告

3. **集計サマリ** (最後)
   - P1 / P2 / P3 件数
   - 検査済セクション一覧 (空セクションがないことを確認)

【出力件数下限】

予想件数: __EXPECTED_COUNT__ の検出能力を期待します。
検出件数が極端に少ない (3 件以下) 場合は再検査してください。

【セクション網羅 mandatory】

検査対象セクション + 「横断」+ 「網羅検証完了」 で計 (N+2) セクション。
各セクションで最低 1 行は明示記載してください (該当なしの場合も「該当なし」と書く / 空セクション禁止)。

【全カテゴリ均等深さ】

本ラウンドで A1-A7 のいずれかを軽く済ませて他に集中することは禁止です。
全カテゴリを前ラウンドと同じ深度で検査してください。
前回検出済の defect 種別も継続してチェックしてください。

【出力形式】

各 finding は以下のフィールドを含めること:

- ID: F-__ROUND_NUM__-001 (連番)
- 重要度: P1 / P2 / P3
- 検査軸: A1 〜 A7 のいずれか
- 該当箇所: ランブック内のセクション番号 + 行番号
- 違和感: 簡潔な記述
- 提案: 修正案 (具体的な diff レベル)
- 矛盾チェック: 前ラウンド方針との整合 (矛盾なし / 矛盾あり: 注記)

最後に集計サマリで P1/P2/P3 件数 + 検査済セクション一覧を出してください。
