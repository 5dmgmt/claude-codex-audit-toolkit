## Codex 監査プロンプト テンプレート (実走証拠ベース v2)
##
## SIFT Phase C R13 ALL PASS で実走した構造を抽象化したもの。
##
## 使い方 (推奨: scripts/codex-audit-prompt-gen.sh 経由):
##   ./scripts/codex-audit-prompt-gen.sh \
##     --runbook docs/foo.md \
##     --runbook-name "Foo ランブック" \
##     --prev-findings "初回" > /tmp/prompt.txt
##   codex exec -s read-only -m gpt-5.5 -c model_reasoning_effort="xhigh" \
##     --output-last-message /tmp/codex-result.md \
##     "$(cat /tmp/prompt.txt)" < /dev/null
##
## 手動で展開する場合:
##   - `sed '/^##/d'` で `##` コメント行を必ず除去してから送信本文化する
##     (除去しないと PREV_FINDINGS が複数箇所に挿入されて prompt が汚染される)
##   - 4 つの placeholder (__SHA__ / __RUNBOOK_NAME__ / __RUNBOOK_PATH__ /
##     __PREV_FINDINGS__) を bash parameter expansion で置換
##   - 監査軸 4-7 の `{...}` 部分は実行前に手で具体的内容に書き換える
##     (派生軸 / 関連参照ファイルは本ランブック固有なので汎用化できない)
##
## 重要:
## - 1 ラウンドで監査対象は 1 ファイルに絞る (多ファイル走査は Codex tool ループの原因)
## - R2 以降は __PREV_FINDINGS__ に「過去ラウンドで反映済の方針」を 30+ 項目蓄積する
## - 蓄積を端折ると矛盾指摘 cascade が再発する
## - codex exec には必ず `< /dev/null` を付ける (stdin 混入防止)

__RUNBOOK_NAME__ (__RUNBOOK_PATH__) を以下 6 軸で網羅的に監査してください。監査対象は作業ツリーの現行ファイルではなく `git show __SHA__:__RUNBOOK_PATH__` のスナップショットとして扱ってください。

> 軸数は 5-7 の範囲で本ランブックに合わせて調整可能です。軸を増減した場合は冒頭の「以下 N 軸」も同期して書き換えてください (Codex は冒頭の軸数宣言を契約として読みます)。

【五月雨防止プロンプト】 — 過去ラウンドとの矛盾 / 同一違和感の深堀り cascade を禁止。
- 過去ラウンドで反映済の修正を pre-condition として受け入れる (例: __PREV_FINDINGS__)。これらを「再考すべき」として再指摘しない
- 同じ違和感の角度を変えた言い換え / 表現の揺れを別件として再指摘しない (1 違和感 = 1 件で集約)
- 1 ラウンドで全 P1/P2/P3 を重複なく列挙し、後続ラウンドでの新規発見を禁止する前提で網羅
- 矛盾する指摘 (前ラウンドで採用された方針を否定する形の指摘) は出す前に「前ラウンド方針との整合」を自己検証してから出す。それでも出すなら明示的に「前ラウンド方針 X との矛盾承知の上で」と注記

【監査軸】 (各軸を 1-2 行で簡潔に。OK/NG 例の三段は不要。本ランブックの性格に合わせて 5-7 軸選ぶ。下記は SIFT 実走で使った既定軸テンプレ — 本ランブック固有の整合軸 (4) に合わせて編集してください)

1. 再現性: 手順を上から実行して同じ結果になるか / 環境差で false FAIL しないか
2. PASS/FAIL の分離: 判定基準が決定的か / 主観表現が混在していないか
3. スコープ整合: タスク表 / 検証手順 / 完了条件 / 進捗トラッカー の対象が完全一致しているか
4. (ランブック固有) 各セクション間の整合: {本ランブック固有の整合軸を 1-2 行で記述。例: §10 と §1-§9 の不可侵領域整合}
5. 環境系: docs/05-env-lint-checklist.md の 14 項目を判定基準として使う (「必ず読む」ではなく「項目に該当する違反があれば指摘」)
6. 設計判断: 明示された前提・制約・完了条件と矛盾しないか / 代替案の好みだけで FAIL にしないか

【監査対象】
- __RUNBOOK_PATH__ (commit __SHA__ のスナップショット)

【関連参照ファイル】 (任意 / 必要なら Read。強制ではない。本ランブック固有のファイルだけを 5 件以内で列挙してください)
- {本ランブック固有の関連ファイル 1}
- {本ランブック固有の関連ファイル 2}

【出力形式】
[P1/P2/P3] [§番号 / 行番号] タイトル
詳細: ...
修正案: ...

最後に [総合判定: ALL PASS or P1×N + P2×N + P3×N] を 1 行で。
